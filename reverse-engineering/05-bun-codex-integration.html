Chapter 05

Bun 기반 AI Code Agent 프로젝트에 통합하는 방법

Open Design을 그대로 옮기는 대신, Codex 전용으로 좁히면 필요한 구현은 명확해집니다. 핵심은 "프롬프트/자산/프로젝트 파일을 준비하고 Codex CLI를 안정적으로 실행한 뒤, HTML artifact를 안전하게 다루는 것"입니다.

최소 모듈 구조 제안

    runs.ts                  # run lifecycle, replayable events, cancel

    runner.ts                # spawn, stdin, env, cwd, timeout

    html-preview/            # srcdoc iframe bridges
    manual-edit/             # DOMParser source patcher

Codex runner contract

실행 args 
codex exec \
  --json \
  --skip-git-repo-check \
  --sandbox workspace-write \
  -c sandbox_workspace_write.network_access=true \
  -C <project-cwd> \
  --model <model> \
  -c model_reasoning_effort="<effort>"

프롬프트는 stdin으로 전달합니다. Open Design 주석에 따르면 최근 Codex CLI는 bare - sentinel을 거부하므로 args에 -를 넣지 않습니다.

환경 변수와 cwd
  • cwd는 반드시 프로젝트 파일 루트. Codex가 생성/편집할 파일 범위를 명확히 합니다.
  • OD_PROJECT_DIR, OD_PROJECT_ID 같은 project hint를 제공하면 media/tooling 확장이 쉬워집니다.
  • CODEX_HOME은 이미지 생성 등 Codex side output 탐색 시 필요할 수 있습니다.
  • active skill sidefile은 .od-skills/<name>로 cwd 안에 복사하는 방식이 Codex sandbox와 가장 잘 맞습니다.

Prompt composer 구현 포인트

Discovery layer

질문 폼, direction picker, TodoWrite, checklist, critique, artifact emission invariant를 가장 앞에 둡니다. 이것이 전체 UX의 핵심이므로 먼저 이식해야 합니다.

PRD metadata layer

기획서 기반 시스템에서는 Open Design의 Project metadata를 확장해 제품 타입, 사용자 역할, 기능 범위, IA, 데이터 모델, auth 요구, 플랫폼 target을 structured block으로 넣습니다.

Skill layer

SKILL.md를 "산출물 종류별 구현 프로토콜"로 사용합니다. 예: prd-to-web-app, prd-to-admin-dashboard, prd-to-mobile-prototype.

Design layer

DESIGN.md는 브랜드 시스템뿐 아니라 도메인 UI 규칙으로 쓰세요. B2B SaaS, fintech, healthcare 등 제품 도메인별 anti-pattern을 넣으면 생성 품질이 안정됩니다.

Codex 전용으로 단순화할 수 있는 설계

단순화
  • agent registry는 Codex 하나만 남기고 model/reasoning 목록만 유지.
  • stream parser는 Codex JSON event만 지원.
  • promptViaStdin은 항상 true.
  • 외부 MCP 전달은 v1에서 제외하고, 필요하면 Codex 설정 파일 기반으로 별도 설계.
  • API fallback provider는 제거. 사용자는 Codex CLI auth를 선행.
그래도 유지해야 할 복잡도
  • 긴 프롬프트/큰 tool result로 인한 run stall 감지.
  • stderr tail과 structured error를 UI에 표시.
  • project cwd와 skill sidefile staging의 권한/경로 검증.
  • artifact가 prose를 HTML로 오인해 저장되는 문제 방지.
  • iframe sandbox와 postMessage source 검증.

구현 우선순위

  1. Codex run lifecycle + SSE + event parser.
  2. Prompt composer + question-form protocol.
  3. SKILL/DESIGN scanner + active resource injection.
  4. Artifact streaming parser + project file 저장.
  5. Sandboxed HTML preview.
  6. Manual edit bridge/source patcher.
  7. Comment/inspect/tweak 고급 기능.
핵심 설계 판단: Bun 시스템이 "기획서 기반 코딩 에이전트"라면 Open Design의 artifact loop를 일반 코드 생성 루프와 분리하세요. PRD → 질문/설계 → HTML prototype은 빠른 feedback surface이고, PRD → 실제 repo 코드 변경은 Codex의 일반 coding workflow입니다. 둘을 같은 iframe/editor로 처리하면 안전 경계가 흐려집니다.
