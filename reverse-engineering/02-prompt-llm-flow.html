Chapter 02

LLM이 항목과 질문을 생성하는 방식

Open Design의 질문/생성 로직은 hard-coded wizard가 아닙니다. Daemon이 system prompt에 규칙과 예시 JSON을 넣고, Codex가 <question-form>, Todo plan, artifact를 순서대로 출력하도록 강제합니다.

Prompt Stack

Layer 0API/plain overrideplain stream adapter일 때 tool call 흉내를 금지. Codex는 JSON event stream이라 보통 미사용.
Layer 1Discovery & philosophy질문 폼, direction picker, TodoWrite, anti-slop, artifact emission invariant.
Layer 2Base designer promptHTML artifact handoff 규약과 designer identity.
Layer 3Memory + DESIGN.md사용자 memory 다음 active design system body를 권위 있는 token source로 삽입.
Layer 4tokens.css/components.htmlOD_DESIGN_TOKEN_CHANNEL=1일 때 구조화된 token/fixture 추가.
Layer 5Craft + SKILL.mdskill이 요구한 craft 문서와 active skill workflow 삽입.
Layer 6Project metadatakind/platform/fidelity/template/animations 등 unknown이면 질문에 포함하라고 지시.
Layer 7Mode-specific addendadeck framework, media contract, Codex imagegen override, MCP guidance.

질문 생성 프로토콜

Turn 1: discovery form

discovery.ts는 새 디자인 작업의 첫 응답을 "짧은 prose 한 줄 + <question-form id=\"discovery\"> + stop"으로 강제합니다.

<question-form id="discovery" title="Quick brief — 30 seconds">
{
  "description": "I'll lock these in before building...",
  "questions": [
    { "id": "output", "label": "What are we making?", "type": "radio", "required": true,
      "options": ["Slide deck / pitch", "Single web prototype / landing", "..."] },
    { "id": "platform", "label": "Target platform", "type": "checkbox", "maxSelections": 4,
      "options": ["Responsive web", "Desktop web", "iOS app", "..."] }
  ]
}
</question-form>

중요한 점은 질문 세트가 고정 UI가 아니라 LLM에게 "tailor the questions"라고 지시된다는 것입니다. 즉 프로젝트 메타데이터가 이미 kind=deck이면 output 질문을 제거하고, speakerNotes가 unknown이면 해당 질문을 추가합니다.

Turn 3+: TodoWrite, self-check, artifact

방향이 확정되면 첫 tool call은 TodoWrite입니다. 계획은 active DESIGN.md, skill asset, layout, placeholder replacement, checklist, 5-dim critique, artifact emission 순서로 구성됩니다.

핵심 invariant: 이 턴에 새 canonical HTML 파일을 쓴 경우에만 <artifact>를 emit합니다. 기존 파일을 편집만 했으면 artifact를 다시 emit하지 않습니다.

Direction Library

directions.ts는 5개 visual direction을 code-level data로 갖고 있습니다. 이 데이터는 질문 폼 카드 렌더링과 prompt 내부 spec block 두 곳에 쓰입니다.

editorial-monocle

serif display, print magazine, paper/ink/red accent. Editorial/Publishing 용도.

modern-minimal

Linear/Vercel 계열, neutral foundation + cobalt accent, software-native.

human-approachable

consumer/education/assistant에 적합. clean neutral + teal accent + generous radius.

tech-utility

GitHub/Datadog 계열. dense table, status pills, mono numerics, engineer-facing UI.

brutalist-experimental

Are.na/Yale 계열. 강한 border, 거대한 serif, mono body, 거의 radius 없음.

질문 폼 파싱과 답변 회수

ParsesplitOnQuestionForms()assistant text에서 <question-form> 블록을 찾아 JSON body를 QuestionForm으로 변환.
RenderQuestionFormViewradio/checkbox/select/text/textarea/direction-cards 렌더링. required와 maxSelections 검증.
SubmitformatFormAnswers()사용자 답변을 [form answers — discovery] 텍스트 메시지로 변환해 다음 Codex turn에 전달.
LockparseSubmittedAnswers()이전 assistant form 바로 다음 user message를 읽어 제출된 폼을 locked state로 렌더링.
[form answers — discovery]
- What are we making?: Dashboard / tool UI
- Target platform: Responsive web, Desktop web
- Who is this for?: Internal planning team
- Brand context: Pick a direction for me

항목 생성 로직의 실제 의미

화면/슬라이드/섹션 항목

코드가 항목을 계산하지 않습니다. Prompt가 "Plan section/slide/screen list with platform variants"라고 요구하고, LLM이 TodoWrite 단계에서 구조를 정합니다. Skill의 references/layouts.md가 있으면 이 구조의 골격을 제한합니다.

메타데이터 기반 질문 축소/추가

renderMetadataBlock()는 이미 알려진 field와 unknown field를 구분합니다. LLM은 unknown field를 turn-1 form에 넣어야 합니다. 이 방식이 "기획서 기반" 시스템에도 가장 유용합니다.

품질 제어

anti-AI-slop checklist, skill checklist P0/P1/P2, 5-dimensional critique가 모두 prompt-level guard입니다. 자동 verifier가 아니라 LLM self-discipline 구조이므로, 이식 시 lint/visual QA로 보강하는 것이 좋습니다.

artifact emission

LLM이 complete HTML document를 <artifact type="text/html">로 감쌉니다. Web은 stream 중에도 preview하고, 종료 후 validateHtmlArtifact()를 통과하면 파일로 저장합니다.

기획서 기반 코딩 에이전트에 적용할 결론: PRD/기획서를 바로 구현하지 말고, 먼저 "unknown field extraction"을 수행해 질문 폼을 생성하세요. 단, 질문 폼 자체를 UI hard-code로 만들기보다 Codex system prompt에 JSON schema와 turn rules를 넣으면 도메인별 질문이 자연스럽게 확장됩니다.
