Chapter 06

역공학 이후 시스템 편입 로드맵

아래 계획은 Open Design 기능을 그대로 복제하는 것이 아니라, 기획서 기반 코딩 에이전트에서 필요한 기능만 Codex 중심으로 흡수하는 작업 순서입니다.

MVP 범위

Phase Plan

Phase 1. Codex execution core
  • Codex adapter를 Open Design에서 분리해 Bun용 runner로 재작성.
  • stdin prompt delivery를 기본값으로 하고 args에 -를 넣지 않음.
  • JSON event parser를 테스트 fixture 기반으로 작성.
  • child close 시 empty-output guard, non-zero exit, signal, structured error를 구분.

완료 기준: fake Codex JSONL fixture와 실제 Codex CLI dry run이 모두 UI text/tool events로 표시됩니다.

Phase 2. Prompt + 질문 폼
  • Open Design discovery layer를 기획서 기반 도메인에 맞게 축약/개조.
  • PRD metadata block 설계: productType, actors, featureScope, platformTargets, dataSources, auth, constraints.
  • 질문 schema는 Open Design과 호환: radio/checkbox/select/text/textarea/direction-cards.
  • 답변 메시지 [form answers — id] 형식을 유지해 Codex가 다음 턴에서 자연어로 읽게 함.

완료 기준: 새 프로젝트 첫 턴에서 질문 폼만 생성하고, 답변 후 Todo/구현 단계로 진입합니다.

Phase 3. Skills/DESIGN registry
  • skills/, design-systems/ root와 user override root를 분리.
  • frontmatter parser와 normalized catalog schema 작성.
  • active skill sidefile staging 구현: project cwd .od-skills/ copy.
  • 기획서 유형별 starter skills 작성: web app, dashboard, mobile, admin, docs.

완료 기준: skill/design 선택이 prompt에 반영되고 Codex가 staged assets/template.html을 읽을 수 있습니다.

Phase 4. HTML artifact preview/store
  • streaming artifact parser를 가져와 code fence/inline code skip 로직 유지.
  • HTML complete document validation 추가.
  • artifact manifest schema 정의: kind, renderer, entry, sourceSkillId, designSystemId, generatedAt.
  • iframe preview에 sandbox shim/base href/deck bridge 최소 구현.

완료 기준: Codex가 emit한 HTML artifact가 stream 중 preview되고 종료 후 project file로 저장됩니다.

Phase 5. HTML editor
  • manual edit bridge와 source patcher 이식.
  • LLM prompt에 data-od-id 생성 규칙 추가.
  • content/style/attributes/html/source 탭 중 content/style/source부터 MVP 구현.
  • undo/redo source snapshot, 외부 변경 충돌 감지.

완료 기준: iframe에서 element 클릭 → panel 수정 → 원본 HTML 저장 → preview 갱신이 가능합니다.

Phase 6. 품질/보안 강화
  • artifact lint: placeholder, lorem ipsum, empty shell, script 위험 패턴 경고.
  • visual regression: Playwright screenshot diff.
  • prompt regression: question-form turn rules와 artifact no-emit invariant 테스트.
  • Codex stall/timeout/large output/permission failure 케이스 UI 처리.

완료 기준: 실패가 silent success로 보이지 않고 사용자에게 원인과 재시도 경로가 표시됩니다.

Risk Register

Codex CLI 버전 차이 Medium

JSON event shape과 flags가 바뀔 수 있습니다. runner는 version을 로깅하고 parser는 unknown event를 raw로 보존해야 합니다.

LLM prompt 불복종 Medium

질문 없이 바로 생성하거나 artifact를 잘못 감쌀 수 있습니다. parser/validator와 retry prompt를 추가해야 합니다.

HTML source patch 손실 Medium

DOMParser 직렬화는 포맷을 보존하지 않습니다. artifact HTML용으로만 사용하고 실제 앱 source는 Codex edit path로 분리합니다.

Untrusted HTML 실행 High

generated HTML은 script를 포함할 수 있습니다. iframe sandbox, no same-origin, postMessage source 검증을 반드시 유지합니다.

프로젝트 파일 덮어쓰기 High

Codex가 기존 파일을 덮을 수 있습니다. prompt에 existing file list와 "do not overwrite"를 넣고, API 저장도 collision 정책을 둡니다.

Skill sidefile 접근 실패 Medium

Codex sandbox에서 repo resource를 못 읽을 수 있습니다. active skill을 cwd 내부로 복사하는 staging을 우선 구현합니다.

Acceptance Criteria

최종 권장 구조: Open Design 기능은 "Design Artifact Workbench"로 모듈화하고, 기존 기획서 기반 코딩 에이전트의 repository editing workflow와 분리하세요. PRD 해석과 질문 폼은 공통으로 쓰되, HTML prototype/editor는 빠른 시각 검증용, Codex code edit는 실제 구현용으로 역할을 나누는 것이 유지보수성이 가장 좋습니다.
