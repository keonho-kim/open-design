핵심 결론

1. Open Design은 자체 LLM 런타임이 아니라 agent orchestration shell입니다

Daemon이 프롬프트를 조합하고 Codex CLI를 headless로 실행합니다. 모델 호출, tool use, 파일 작성은 Codex가 담당하고, Open Design은 실행 디렉터리, system prompt, SSE 이벤트, artifact 파싱/저장을 담당합니다.

시스템 구조 보기

2. 질문 생성은 프론트엔드 기능이 아니라 system prompt protocol입니다

<question-form>은 LLM이 생성하는 XML-ish 블록이고, 웹이 JSON body를 파싱해 폼으로 렌더링합니다. 답변은 다시 [form answers — id] 텍스트로 다음 턴에 들어갑니다.

질문/LLM 흐름 보기

3. HTML 에디터는 DOM bridge + source patcher 조합입니다

프리뷰 iframe은 sandboxed srcdoc입니다. selection/comment/inspect/manual-edit bridge 스크립트를 주입하고, 수동 편집은 DOMParser로 원본 HTML을 수정한 뒤 프로젝트 파일 API에 저장합니다.

HTML 에디터 분석 보기

4. Bun 프로젝트에 이식하려면 6개 경계가 필요합니다

리소스 레지스트리, prompt composer, Codex runner, stream normalizer, artifact parser/store, iframe/editor runtime을 독립 모듈로 분리하면 됩니다. OpenAI Codex만 대상으로 하면 provider abstraction은 크게 줄일 수 있습니다.

복사된 원본 자산 위치

원본 SKILL.md, DESIGN.md, AGENTS.md는 상대 경로를 유지한 채 assets/source-files/ 아래 복사했습니다. 예: web-prototype/SKILL.md, openai/DESIGN.md.

주요 분석 원본 코드
  • apps/daemon/src/server.ts: chat run 조립, Codex 실행, cwd/스킬 staging.
  • apps/daemon/src/prompts/system.ts: system prompt composer.
  • apps/daemon/src/prompts/discovery.ts: 질문 폼/방향 선택 프로토콜.
  • apps/daemon/src/runtimes/defs/codex.ts: Codex CLI adapter.
  • apps/web/src/components/ProjectView.tsx: artifact parser와 저장.
  • apps/web/src/runtime/srcdoc.ts: sandbox iframe bridge 주입.
  • apps/web/src/edit-mode/source-patches.ts: manual edit source patching.
