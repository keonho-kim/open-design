Chapter 04

SKILL.md와 DESIGN.md 로딩/주입 구조

Open Design의 품질은 코드보다 파일 기반 지식 자산에 크게 의존합니다. SKILL.md는 산출물 종류별 workflow이고, DESIGN.md는 브랜드/시각 시스템 token source입니다.

Skill 로딩

1. scan + frontmatter parse

skills.ts의 listSkills()는 root 배열을 순서대로 훑고 각 하위 폴더의 SKILL.md를 읽습니다. 첫 번째 root가 user root라서 id collision 시 사용자 스킬이 built-in을 shadow합니다.

frontmatter.ts는 dependency 없는 YAML subset parser입니다. scalar, flat array, block literal 정도만 지원하므로 복잡한 YAML은 피해야 합니다.

3. sidefile이 있으면 skill root preamble 추가

스킬 폴더에 assets/, references/, scripts/ 등이 있으면 body 앞에 "Skill root absolute/cwd-relative" 정보를 추가합니다. 이것이 Codex가 assets/template.html, references/checklist.md를 실제 파일로 읽게 만드는 장치입니다.

Design System 로딩

1. catalog summary

design-systems.ts는 design-systems/*/DESIGN.md를 스캔해 id, title, category, summary, swatches, surface, body를 만듭니다.

  • title: 첫 H1에서 추출하고 "Design System Inspired by" 같은 prefix 제거.
  • category: H1 아래 > Category: blockquote.
  • swatches: markdown의 hex color token을 regex로 뽑아 bg/support/fg/accent 4개 선택.
  • surface: > Surface: 없으면 web.
2. prompt injection

선택된 DESIGN.md body는 composeSystemPrompt()에서 ## Active design system 블록으로 들어갑니다. 지시 문구는 색/타이포/간격/component rule의 권위 있는 source로 취급하라고 명시합니다.

OD_DESIGN_TOKEN_CHANNEL=1이면 같은 폴더의 tokens.css, components.html도 추가로 읽어 CSS token contract와 reference fixture로 삽입합니다.

3. schema

문서상 가이드는 9개 numbered section을 요구합니다. 실제 scanner는 H1/category/swatches 중심으로 동작하지만, prompt 품질과 skill grep 의존성을 위해 9-section 형식 유지가 중요합니다.
